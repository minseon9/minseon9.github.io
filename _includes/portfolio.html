리뷰(평가) 도메인 개발 / 운영

리뷰/평가 도메인의 legacy 리팩토링, 제품 고도화 등 고객 문제 해결

역할: HR 도메인 중 리뷰/평가 도메인에서의 고객 문제 해결
기술 스택: Python, Django, MySQL, Amplitude, Hotjar, DDD
문제 정의
상대 평가, 리뷰 결과지 고도화, 정량평가 등 리뷰/평가 도메인을 개발했습니다.
고객 피드백과 소통 등을 통해 고객의 진짜 문제를 이해하고 도메인 개념을 정립하여 비즈니스 임팩트를 만들어냈습니다.
문제 해결
  • 직접 Figma를 활용해 기능 복잡도를 줄이는 사용자 흐름을 설계하고, 팀에 솔루션을 제안했습니다.
    • 하나의 평가 프로젝트를 생성하는 시점에, 특정 기능을 ON/OFF함에 따라 다른 설정값들이 영향을 받고, 평가를 진행하는 모든 화면에서 분기가 일어나 어드민과 사용자 모두 이해하기 어려운 UX를 제공했습니다.
    • 이를 기본값의 형태로 제공하므로써, 일괄된 플로우로 복잡도도 낮으며 해당 기능을 사용하지 않는 고객 또한 큰 학습 비용없이 사용할 수 있음을 Figma 시안을 통해 보였습니다.
      이 제안은 비즈니스 팀과 고객 인터뷰에서도 긍정적인 피드백을 받아 반영 서비스 반영까지 이어졌습니다.
  • VoC와 고객 인터뷰를 통해 요구사항을 분석하고, Hotjar 설문 및 Amplitude 데이터 분석을 통해 개선 사항을 도출했습니다.
    • 기능 고도화 설정이 복잡해져 빠른 설정이 어려워지고, 특히 규모가 작은 고객사에게는 불필요한 기능으로 오히려 불편하다는 피드백을 수용하였습니다.
      다양한 고객 규모를 지원 등 솔루션 차원에서의 고려 사항을 체크리스트로 만들어 높은 제품 퀄리티를 유지하도록 했습니다.
    • 모바일 앱에서 리뷰 작성 기능을 개발하는 과정에서, 리소스 제한으로 지원하지 못한 케이스에 대해 Amplitude 데이터를 수집하여 해당 기능의 필요성과 우선순위를 파악했고, 향후 모바일 개발 scope을 정의하는데 활용했습니다.
성과 & 배움
  • SIP(사내 비즈니스 임팩트 지표) 총 240점 차감
    • 리뷰 도메인 전체 점수의 약 25% 차지하며, 이는 기능 고도화가 실질적인 고객 만족과 문제 해결로 이어졌음을 의미합니다.
  • 결과를 구현해내는 개발자가 아닌 문제 정의 단계부터 참여해 솔루션을 제시하며, 모든 과정에서 팀과 함께 일하는 것을 중요시합니다.

제품 성장에 따른 API 성능 최적화

역할: 서버 사이드 페이지네이션 구현 및 제품 전반 성능 병목 개선
기술 스택: Python, Django, MySQL, Datadog
문제 정의
개발자는 문제가 발생할 수 있는 지점들을 미리 파악하고, 비즈니스 상황에 맞춰 적합하게 스스로 개선해야합니다.
제품 성장에 따라 예상되는 성능 문제를 사전에 개선하고 비즈니스 임팩트로 연결했습니다.
문제 해결
  • 리뷰 목록에 카테고리를 지원하는 프로젝트에서 성능 문제를 예상하고, 서버 사이드 페이지네이션을 공통 모듈로 만들어 개선했습니다.
    • 비즈니스가 성장함에 따라 성능 문제가 충분히 예상되는 지점이었습니다.
      프로젝트의 기간을 준수하면서 서버 사이드 페이지네이션을 공통 모듈로 만들어 성능 문제를 사전에 방지했습니다.
    • '나중에 고치자'는 태도를 가능한 지양합니다.
  • 데이터 양을 고려해 검색 결과에 따라 필터 옵션이 동적으로 변하도록 정책을 제안하였고, 필터 결과가 없는 필터는 제외하도록 구현했습니다.
  • 해당 프로젝트 마무리 후 다음 분기에 약 2만 명 규모의 글로벌 기업이 구독하였고, 서버 사이드 페이지네이션을 제품 전반에 적용해 성능 상 병목이 되는 지점들을 빠르게 개선했습니다.
성과 & 배움
  • 구성원 조회 기준 성능 18.2초 → 2.2초로 단축
  • 성능 개선 및 구성원 2만 명 규모의 글로벌 기업 도입
    • 병목을 사전에 제거하여 VoC 없이 리뷰를 마무리했습니다.
    • 서비스에 발생할 수 있는 문제를 예측하고, 상황이 발생했을 때의 리스크와 기회 비용을 잘 따져 개선하는 것 또한 개발자의 몫입니다.
      지금 당장 시간을 조금 더 쓰더라도, 결국에는 우리의 속도를 올리는 일감들을 잘 판단하고 수행하는 것이 얼마나 중요한 일인 지 배웠으며, 비즈니스에 성과로 이어진 경험입니다.

관리자가 다른 사용자의 계정으로 로그인할 수 있는 기능(Master Login) 구현

역할: 읽기 전용 계정 미들웨어 개발 및 조회 불가능 정보 마스킹 처리 개발
기술 스택: Python, Django, MySQL
문제 정의
관리자 계정이 특정 사용자 계정으로 로그인해 사용자가 실제로 마주하는 화면을 확인할 수 있도록 Master Login 기능을 설계 및 구현했습니다.
권한 인증 구조를 재설계하며, 발생 가능한 보안·운영 리스크를 체계적으로 관리하며 프로젝트를 마무리하는데 집중했습니다.
session 관리와 민감 데이터 접근 제어 같은 기술적 문제도 해결하며, 어드민이 안정적으로 평가를 운영할 수 있도록 구현했습니다.
문제 해결
  • 기존의 데모 계정을 읽기 전용 계정으로 추상화했고, 인터페이스 기반으로 동작하도록 미들웨어를 개선했습니다.
    • 민감한 부문에 대한 개선으로 데모 계정과 미들웨어 등의 기존 구현을 기반으로 테스트 케이스를 보강하고 변경 사항에 따른 리스크를 관리했습니다.
  • 마스터 로그인 세션 기반 권한 인증 구조를 설계하고, 동시 요청(racing)과 세션 충돌을 방지하도록 구현했습니다.
    • 마스터 로그인의 로그아웃과 다른 API의 racing condition에서, session id가 다시 덮여쓰이는 경우가 발생했습니다.
    • 로그아웃을 Redirect로 변경해 racing하는 XHR 요청을 무시하도록 문제를 해결했습니다.
  • 마스터 로그인 상태에서 조회할 수 없는 민감 데이터는 마스킹해 접근이 불가능하도록 했습니다.
    • 전체 제품에서의 정책들이 정리되어 있지 않았고, 먼저 코드 기반으로 정리해 정책을 고안했습니다.
    • 마스터 로그인 상태에서 볼 수 없음을 응답 객체에 정의하고, 응답값을 serialize할 때 마스킹이 되도록 구현했습니다.
  • 마스터 로그인 후 호출된 API에 대한 보안 감사를 남기는 등 보안 이슈를 관리했습니다.
성과 & 배움
  • Master Login 기능 도입 후 2개월간 고객사 30% 이상 사용, Hotjar 설문 결과 만족도 4/5점 초과
  • 전체 제품에 적용하는 프로젝트로 사이드이팩트와 오류 없이 프로젝트 완료

제품 별 해야 할 일, 알림 개편 등 홈 화면 개편

역할:제품 별 해야 할 일(Task) 설계 및 구현
기술 스택: Python, Django
문제 정의
사용자가 리뷰, 1:1, 목표, 서베이 등 다양한 제품에서 해야 할 일을 한 눈에 확인하기 어렵다는 문제가 있었습니다.
이에 따라 홈 화면 개편 프로젝트의 일환으로, 제품 전반에 공통 적용 가능한 해야 할 일(Task) 도메인의 설계 및 구현을 담당했습니다.
MSA, 이벤트 기반 구조와 같은 infra 고민도 함께 했던 프로젝트입니다.
문제 해결
  • 알림과 해야 할 일(Task) 도메인을 별도 서버로 분리하며, 인프라적 관점에서의 확장성과 안정성 확보, 팀 차원에서의 운영·관리 리스크 최소화라는 두 가지 관점에서 아키텍처 설계를 고민했습니다.
    • 알림/할 일(Task) 도메인을 별도 서비스에서 처리하고 Event Driven 구조에 적합하다고 판단해 아키텍처를 설계했습니다.
    • 단, 당시 팀의 규모와 비즈니스 상황상 MSA를 유지·보수할 여력이 없다고 판단해, 기존 구조 내에서의 클린 아키텍처에 집중했습니다.
  • Django ORM의 한계를 극복하고 도메인 계층을 분리하기 위한 Task 아키텍처를 설계했습니다.
    • Django ORM 모델은 Django Query와 강하게 결합되어 있어, 순수한 도메인 모델로의 동작이 어렵고, 테스트 시 mocking 또한 복잡합니다.
    • 계층별 역할을 분리하여 ORM에서는 TaskableMixin으로 Task 지원을 명시하고, 도메인 계층에서는 Task[T: TaskableMixin] 추상 모델로 비즈니스 로직을 처리하도록 하여 두 계층 간의 의존성을 완전히 분리했습니다.
    • 결과적으로 모든 도메인에서 인터페이스에 따라 Task를 정의하는 것으로 해야 할 일 관리가 가능하도록 했으며, 도메인 로직의 단위 테스트가 DB 의존성 없이 가능해졌습니다.
성과 & 배움
  • 홈 화면 전면 개편과 함께 사용자별 해야 할 일을 일관되게 제공해, 제품 내 접근성과 사용성을 향상했습니다.
  • MSA 전환 여부를 기술적 관점이 뿐 아닌 비즈니스 상황과 운영 리스크를 고려해 의사결정한 경험을 쌓았습니다.

다수 TF 진행 및 비즈니스 임팩트 창출

Magic Link 기반 비로그인 접근 지원

역할: 비로그인 인증 설계 및 구현 전반
기술 스택: Python, Django, MySQL, Kafka, NHN SMS, SendGrid
문제 정의
구성원의 가입 절차 없이 사용하길 원하는 고객사의 경우, 제품 사용이 어려웠습니다. 비로그인 권한 인증을 구축하고, 특정 제품을 사용할 수 있도록 지원한 프로젝트입니다.
타 스쿼드의 업무를 extra mile로 지원한 프로젝트로 인증 설계부터 구현 전반을 담당하여 진행했습니다.
비로그인 상태에서도 접근할 수 있는 Magic Link 기능을 구현하며, 기존 권한 인증 구조와의 사이드 이펙트와 같은 리스크를 없애는데 초점을 맞췄습니다.
문제 해결
  • 인증 로직은 토큰 기반으로 설계하였으며, 유효 시간, 토큰 무효화, 특정 엔드포인트에만 접근 허용 등 보안 요구사항을 고려해 구현했습니다.
    • 인증 상태, 비로그인 상태에서 기존 엔드포인트 접근 시 발생할 수 있는 모든 리스크를 테스트 케이스와 POC를 통해 프로젝트 초반에 사전 관리했습니다.
    • Django의 미들웨어 특성상 기존 권한 인증과 비로그인 인증이 혼재될 때의 문제를 발견하고, 비로그인 인증은 decorator를 적용한 API에 한정하도록 구조를 개선했습니다.
  • Magic Link는 SMS와 이메일로 발송하도록 구성했습니다.
    • 비로그인 상황에서 접근성을 높이기 위해 SMS도 지원해야함을 제안했습니다. 고객이 겪게될 문제를 항상 고려합니다.
    • Outbox Pattern을 사용해 SMS/Email 발송 이벤트를 생성하고, Kafka 기반 비동기 처리하였습니다.
성과 & 배움
  • 엔터 프라이즈 기업 도입 및 도입 기업의 Magic Link 서베이 참여율 80% 상회
    • SMS 지원으로 해당 기업이 이전에 다른 플랫폼을 통해 서베이를 했을 때의 참여율을 상회하는 결과를 얻었습니다. 고객의 문제를 해결하기 위해 몰두하며, 제품의 퀄리티를 스콥 아웃하지 않습니다.
  • 스쿼드 업무 디스커버리 기간동안 타 스쿼드 업무를 병행했던 프로젝트입니다.팀 전체의 비즈니스 임팩트를 항상 고려하며, 잘 모르는 분야도 도전하고 성취하는 것을 즐깁니다.
  • 테스트 케이스와 설계에 대한 피드백을 수시로 요청했습니다. 그 결과 문제가 될 지점들을 빠르게 발견하는 등 리스크를 관리하며 프로젝트를 마무리했습니다.

글로벌 시장 대응을 위한 국제화(i18n) 및 번역 자동화 구축

역할: 백엔드 국제화 대응 및 번역 자동화 구축
기술 스택: Python, Django, MySQL, Crowdin, GitHub Actions
문제 정의

제품의 국제화 및 번역 자동화를 구축한 프로젝트이며, 한국어와 영어, 일어를 지원했습니다.
개발자가 개발을 하면서 infra 차원에서 고민할 것이 없도록 구현했습니다.
문제 해결
  • 백엔드 코드 전반에 국제화를 적용했습니다.

기술 스택: Rust, Rocket, Node.js, Express, MySQL
문제 정의
Multi-Party Computation(MPC)은 연산에 참여하는 각 party의 input이 들어나지 않은 채로 모든 party가 동일한 결과들 얻도록 하는 연산 방식입니다.
지갑 서비스에서 가장 중요한 비밀값은 private key입니다. 대게의 경우 multi-sig를 사용해 private key를 분산 보관합니다. 하지만 서명하거나 키를 생성할 때 private key 원형이 노출되는 single point of failure가 존재한다는 단점이 있습니다.
이런 보안 이슈를 극복하기 위해 MPC를 활용합니다.
문제 해결
  • 논문과 오픈 소스를 리서치하고, MPC 연산을 사용해 블록체인 트랜잭션에 서명하고 전송하는 테스트 진행했습니다.
  • 제품에 적용하기 위한 아키텍쳐와 데이터베이스 설계, 구현까지 PoC 전반을 담당했습니다.
    • Rust Rocket의 오픈 소스를 기반으로 2 party 형식의 peer-2-peer 서버를 구현하고, 지갑 서비스와 연동해 제품에 적용까지 PoC를 마무리했습니다.
성과 & 배움
  • 리서치부터 오픈소스 테스트와 제품 설계까지 모든 과정을 주도한 프로젝트로 실질적인 문제 해결 경험을 쌓았습니다.
