기술 스택: Python, Django, MySQL, Amplitude, Hotjar, k6
프로젝트 개요
리뷰 / 평가 도메인의 Legacy 리팩토링과 제품 고도화, 성능 문제 개선, 고객 피드백 기반 기능 개선 등의 다양한 업무를 수행했습니다.
고객사마다 평가 정책은 모두 상이합니다. 고객의 진짜 문제를 이해하고 정립하여 비즈니스 임팩트를 만들어낸 경험입니다.
문제 해결
상대 평가 지원, 결과지 고도화, Master Login(관리자가 특정 사용자 권한으로 접근) 지원 등 주요 기능을 전면 개선하고 고도화했습니다.
정량 평가 지원을 위해 리뷰 도메인을 전반적으로 개편하는 프로젝트를 진행했습니다.
  • 직접 Figma를 활용해 솔루션을 제안하고 팀원들과 논의하기도 합니다. 단순히 결과물을 구현하는 개발자가 아닌 문제 해결 중심의 협업을 지향합니다.
  • 해당 제안은 비즈니스 팀으로부터도 긍정적인 피드백을 받아 개발 과정에 반영되었습니다.
리뷰 목록에 카테고리를 지원하는 프로젝트에서 서버 사이드 페이지네이션을 도입하고, 공통 모듈로 개발하여 재사용성을 확보했습니다.
  • Boy Scout Rule을 지향하며, '나중에 고치자'는 태도를 지양합니다.
  • 해당 프로젝트는 페이지네이션과는 무관한 프로젝트입니다. 하지만 이 프로젝트를 진행하면서 개선함으로써, 대규모 엔터프라이즈 고객 도입 시 성능 병목을 빠르게 개선했습니다.
개발 전 단계에서 VoC와 고객 인터뷰를 통해 요구사항을 분석하고, 개발 이후에는 Hotjar 설문 및 Amplitude 데이터 분석을 통해 개선 사항을 도출했습니다.
  • 리뷰 결과지 고도화 당시, 일부 고객에게는 불필요한 기능으로 오히려 불편하다는 피드백을 수용하였고, 솔루션 개발 시 즉시 반영했습니다.
  • 모바일 앱에서 리뷰 작성 기능을 개발하는 과정에서, 공수(리소스) 제한으로 지원하지 못한 케이스에 대해 Amplitude 데이터를 수집하여 해당 기능의 필요성과 우선순위를 파악했고, 향후 모바일 개발에 반영할 수 있도록 준비했습니다.

Magic Link 기반 비로그인 접근 지원

역할: 비로그인 인증 설계 및 구현 전반
기술 스택: Python, Django, Kafka, NHN SMS, Sendgrid
프로젝트 개요
구성원의 가입 절차 없이 사용하거나, 구성원의 익명성 등 보안이 중요한 고객사는 제품 사용이 어려웠습니다.
이런 니즈의 증가로 비로그인 권한 인증을 구축하고, 특정 제품을 사용할 수 있도록 지원한 프로젝트입니다.
문제 해결
제품을 회원 가입 없이 사용하거나, 익명성과 보안이 중요한 상황에서도 접근 가능한 기능을 제공하기 위해 Magic Link 기반의 비로그인 인증 방식을 설계하고 구현했습니다.
  • 인증 로직은 토큰 기반으로 설계하였으며, 유효 시간 제한, 토큰 취소 처리, 특정 엔드포인트에만 접근 허용 등 보안 요구사항을 고려한 정책을 함께 구현했습니다.
Magic Link는 SMS 및 이메일로 발송되며, NHN SMS와 Sendgrid를 연동하여 다양한 채널에서 메시지를 발송할 수 있도록 구성했습니다.
  • 메시지 발송은 Outbox Pattern을 사용해 Kafka 기반 비동기 처리하였습니다.
인증 처리 미들웨어, 데코레이터 등 모든 구현에 단위 테스트를 작성했고, PoC와 빠른 리뷰를 통해 프로젝트 전반의 리스크를 관리했습니다.
성과 & 배움
  • 엔터프라이즈 기업 도입 및 세일즈 확대에 기여
  • 도입 기업의 Magic Link 서베이 참여율 80% 상회
  • 본인의 스쿼드 업무 디스커버리 기간인 약 3주동안 타 스쿼드 업무를 지원해준 프로젝트입니다.팀 전체의 비즈니스 임팩트를 항상 고려하며, 잘 모르는 분야도 도전하고 성취하는 것을 즐깁니다.
  • 권한 인증과 관련된 코드 리뷰와 설계에 대한 피드백을 수시로 요청했습니다. 그 결과 문제가 될 지점들을 빠르게 발견하는 등 리스크를 관리하며 프로젝트를 마무리했습니다. 적극적인 소통과 개발자로서 위험을 관리하는 법을 배웠습니다.

구독 정책 고도화 및 특약 지원

역할: 특약 구독 설계 및 구현. QA 효율화를 위한 사내 시스템 구축
기술 스택: Python, Django, Appsmith
프로젝트 개요
기존 구독 시스템은 고객사별 특약을 반영하기 어려운 구조로 되어 있어,
특약 계약이 필요한 경우 현금 결제와 계약서 수기 처리, 세금계산서 수동 발행 등 수작업 업무가 반복적으로 발생하고 있었습니다.
제품 내에서 특약 조건을 유연하게 수용할 수 있도록 구독 도메인을 고도화한 프로젝트입니다.
문제 해결
고객사별 특약을 반영할 수 있도록 구독 로직을 재설계하고, Appsmith를 활용해 특약을 발행하도록 구현했습니다.
구독 상황별 테스트를 위해 결제 도메인을 Mocking하는 내부 QA용 도구를 개발했습니다.
  • 이전까지는 개발자가 직접 계정 상태를 수작업으로 조작해주어야 했으나, 사내 시스템을 구현해 QA 담당자가 직접 다양한 구독 시나리오를 테스트할 수 있도록 간편하게 구성했습니다.
  • 사내 시스템은 Django Template을 활용한 MVC 패턴으로 개발했으며, Cursor 등 AI 기반 개발 보조 툴을 활용해 빠르게 구축했습니다.
구독 상태에 대한 이상 징후나 실패 케이스를 CloudWatch 및 Sentry 기반 모니터링했습니다.
성과 & 배움
  • 기존에 수작업으로 진행되던 특약 기반의 결제 및 관리 프로세스를 제품 내에서 자동화함으로써, 운영 공수 대폭 절감 및 QA 효율화
  • 특약 구독을 제품에서 직접 지원함으로써, 계약부터 결제까지의 전체 플로우를 간소화함으로 업무 효율화
  • 불편함이나 비효율을 개선해 팀 전체의 업무 효율을 높이는 것을 지향합니다. 개발자의 역량으로 팀 전체의 효율을 올릴 수 있음을 배웠습니다.

SageMaker 기반의 LLM 인프라를 구축하여, 타 개발자들이 쉽게 LLM을 테스트하고 활용할 수 있도록 환경을 마련하고 문서화까지 완료했습니다.
사내 AI 해커톤을 기획 및 개최해, 실제 서비스 시나리오에 LLM을 적용한 기능을 실현해보는 과정을 진행했습니다.
성과 & 배움

Solana(솔라나) 입출금 지원 및 입금 아키텍처 개선

역할: 설계 및 개발 리드 (아키텍처 설계, DB 설계, 구현 전반)
기술 스택: Typescript, NestJS, ElasticCache(Redis), BullMQ
프로젝트 개요
솔라나는 평균 TPS가 3000이 넘어가는 블록체인 플랫폼입니다.
한 transaction에는 여러 event가 발생하므로 블록 당 처리해야하는 event는 약 10,000건이 넘어갑니다.
기존 입금 처리 로직으로는 cron job이 일정 주기마다 쌓인 블록을 처리하는 방식으로 솔라나와 같이 처리량이 매우 많아진 사항을 처리할 수 없습니다.
솔라나의 높은 TPS와 이벤트 처리량을 감당하기 위해 병렬 처리 아키텍처로 전환했습니다.
문제 해결
두 가지 방안을 비교했으며, 각각의 장단점은 다음과 같습니다:
  • 모듈러 연산 기반 Job 분산: 처리할 블록을 job 단위로 분할하는 방식으로, 구현이 단순하고 기존 아키텍처와 유사하다는 장점이 있습니다. 하지만 DB와의 결합도가 높고 job 수 변경 시 도메인과 DB 양쪽에 수정이 필요하며, 특정 job 실패 시 해당 모듈러에 속한 블록 전체 처리가 지연된다는 단점이 있습니다.
  • Message Queue 기반 처리 (채택): 아키텍처 변경에 대한 고려가 더 필요했지만, 확장성과 장애 대응이 유연하다는 장점이 있습니다. 실패한 작업만 재처리할 수 있다는 것 또한 큰 장점입니다.
최종적으로 BullMQ를 활용해 블록을 병렬 처리하도록 구현했습니다. 이미 사용 중인 ElasticCache를 기반으로 구축해 인프라 부담을 줄였으며, 처리할 블록 정보를 가져오는 통신 레이어를 추상화해 의존을 역전 시켜 향후 인프라 변경에 용이하도록 설계했습니다.
성과 & 배움
  • Message Queue 기반 병렬 처리 구조로 전환하는 비교적 큰 변화였음에도 2주 내에 완료
    • 초기 설계 단계에서 입금 도메인과 블록 처리 도메인을 명확히 분리해 두었기 때문에, MQ 도입 시에도 최소한의 수정으로 전환
  • 기존 로직에서는 블록 처리 지연으로 서비스가 불가능했지만, 개선 후에는 블록 생성 후 1초 이내 입금 반영이 가능한 수준으로 개선

Multi-Party Computation Based Threshold Signature

역할: 논문 & 오픈 소스 리서치, 제품 PoC 설계 및 구현, 특허 등록
기술 스택: Typescript, NestJS, ElasticCache(Redis), BullMQ
프로젝트 개요
Multi-Party Computation은 연산에 참여하는 각 party의 input이 들어나지 않은 채로 모든 party가 동일한 결과들 얻도록 하는 연산 방식입니다.
지갑 서비스에서 가장 중요한 비밀값은 private key입니다. 대게의 경우 multi-sig를 사용해 private key를 분산 보관합니다. 하지만 서명하거나 키를 생성할 때 private key 원형이 노출되는 single point of failure가 존재한다는 단점이 있습니다.
이런 보안 이슈를 극복하기 위해 MPC를 활용합니다.
문제 해결
처음 한 달간은 논문을 한줄 한줄 읽어가며 무엇을 하려고 하는지 이해했습니다. 오픈 소스를 찾아 논문과 비교하며 어려운 수식들이 코드로 구현되어 있는 것들을 확인하고 MPC 연산을 테스트 했습니다.
그리고 Rust Rocket의 오픈 소스를 기반으로 2 party 형식의 peer-2-peer 통신 서버를 구현하고, 지갑 서비스와 연동해 제품에 적용까지 PoC를 마무리했습니다.

  • 리서치부터 오픈소스 테스트와 제품 설계까지 모든 과정을 주도한 프로젝트로 실질적인 문제 해결 경험을 쌓았습니다.
