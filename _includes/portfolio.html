리뷰(평가) 도메인 개발 / 운영 / 유지보수

리뷰/평가 도메인의 legacy 리팩토링, 제품 고도화 등 리뷰(평가) 고객 문제 해결

역할: HR 도메인 중 리뷰/평가 도메인에서의 고객 문제 해결
기술 스택: Python, Django, MySQL, Amplitude, Hotjar
프로젝트 개요
상대 평가, 리뷰 결과지 고도화, 정량평가 등 리뷰/평가 도메인을 유지보수하는 다양한 업무를 수행했습니다.
고객사마다 평가 정책은 모두 상이합니다. 고객 피드백과 소통 등을 통해 고객의 진짜 문제를 이해하고 도메인 개념을 정립하여 비즈니스 임팩트를 만들어낸 경험입니다.
문제 해결
  • 직접 Figma를 활용해 기능 복잡도를 줄이는 사용자 흐름을 설계하고, 팀에 솔루션을 제안했습니다.
    • 새로운 기능의 사용 유무에 따라 제품 전반에 분기가 일어나는 것은 고객의 학습 비용과 제품 복잡도를 올리는 것은 물론 앞으로의 제품 개발에 걸림돌이 된다고 판단했습니다. 본인이 생각하기에 일괄된 플로우로 복잡도가 낮고, 기능을 사용하지 않는 고객도 쉽게 넘어갈 수 있는 솔루션을 Figma로 그려 제안했습니다.
    • 해당 제안은 비즈니스 팀과 고객 인터뷰에서도 긍정적인 피드백을 받아 개발 과정에 반영되었습니다.
      단순히 결과물을 구현하는 개발자가 아닌 고객 문제 해결 중심의 협업을 지향합니다.
  • 개발 전 단계에서 VoC와 고객 인터뷰를 통해 요구사항을 분석하고, 개발 이후에는 Hotjar 설문 및 Amplitude 데이터 분석을 통해 개선 사항을 도출했습니다.
    • 리뷰 결과지 고도화 당시, 일부 고객에게는 불필요한 기능으로 오히려 불편하다는 피드백을 수용하였고, 솔루션 개발 시 반영했습니다.
    • 모바일 앱에서 리뷰 작성 기능을 개발하는 과정에서, 리소스 제한으로 지원하지 못한 케이스에 대해 Amplitude 데이터를 수집하여 해당 기능의 필요성과 우선순위를 파악했고, 향후 모바일 개발에 반영할 수 있도록 준비했습니다.
성과 & 배움
  • SIP(사내 비즈니스 임팩트 지표) 총 240점 차감
    • 리뷰 도메인 전체 점수의 약 25% 차지. 이는 기능 고도화가 실질적인 고객 만족과 문제 해결로 이어졌음을 의미합니다.
  • 문제를 해결하는 과정에서 팀과 고객과 적극적으로 소통합니다. 좋은 제품을 만들기 위해 이런 소통을 통해 문제를 잘 정의하고 정립하는 것의 중요성을 배웠습니다.

제품 성장에 따른 API 성능 최적화

역할: 서버 사이드 페이지네이션 구현 및 제품 전반 성능 병목 개선
기술 스택: Python, Django, MySQL, Datadog
프로젝트 개요
제품이 성장함에 따라 성능 문제가 예상되는 지점을 미리 개선하고, 비즈니스 임팩트로도 이어졌던 경험입니다.
문제 해결
  • 리뷰 목록에 카테고리를 지원하는 프로젝트에서 Django QuerySet을 확장해 서버 사이드 페이지네이션을 공통 모듈로 구현하였고, 성능 문제를 개선했습니다.
    • Boy Scout Rule을 지향하며, '나중에 고치자'는 태도를 가능한 지양합니다.
    • 페이지네이션을 꼭 구현해야하는 프로젝트는 아니었지만, 비즈니스가 성장함에 따라 성능 문제가 충분히 예상되는 지점이었습니다.
      프로젝트의 기간을 준수하면서 서버 사이드 페이지네이션을 공통 모듈로 만들어 성능 문제를 사전에 방지했습니다.
  • 해당 프로젝트 마무리 후 다음 분기에 약 2만 명 규모의 글로벌 기업이 구독하였고, 서버 사이드 페이지네이션을 제품 전반에 적용해 성능 상 병목이 되는 지점들을 빠르게 개선했습니다.
성과 & 배움
  • 구성원 조회 기준 성능 18.2초 → 2.2초로 단축
  • 성능 개선 및 구성원 2만 명 규모의 글로벌 기업 도입
    • 대규모 기업에서도 VoC 없이 안정적으로 사용 가능하도록 병목을 사전에 제거
    • 서비스에 발생할 수 있는 문제를 예측하고, 상황이 발생했을 때의 리스크와 기회 비용을 잘 따져 개선하는 것 또한 개발자의 몫입니다.
      지금 당장 시간을 조금 더 쓰더라도, 결국에는 우리의 속도를 올리는 일감들을 잘 판단하고 수행하는 것이 얼마나 중요한 일인 지 배웠으며, 비즈니스에 성과로 이어진 경험입니다.

관리자가 다른 사용자의 계정으로 로그인할 수 있는 기능(Master Login) 구현

역할: 읽기 전용 계정 미들웨어 개발 및 조회 불가능 정보 마스킹 처리 개발
기술 스택: Python, Django, MySQL
프로젝트 개요
리뷰/평가는 다루는 정보가 민감한 만큼, 고객사의 인사팀은 사용자의 이슈를 재현하거나,사용자가 실제로 마주하는 화면을 확인해야합니다.
관리자 계정이 특정 사용자 계정으로 로그인해 수 있도록 Master Login 기능을 설계 및 구현했습니다.
문제 해결
  • 읽기만 가능한 데모 계정과 마스터 로그인 계정의 공통 구조를 추상화해 구조를 개선했습니다.
    • 공통 구조를 추상화해 읽기 전용 계정 mixin을 구현했고, 미들웨어에서는 추상화된 읽기 계정으로 특정 조건의 endpoint를 blacklist하도록 구조를 개선했습니다.
    • 본격적으로 업무를 시작하기 전, 리팩토링을 진행해 방향성을 빠르게 검토했습니다.
  • 관리자 계정은 볼 수 없는 비공개 정보 마스킹 처리와 마스터 로그인 후 호출된 API에 대한 보안 감사를 남기는 등 보안 이슈를 관리했습니다.
  • 구성원 관리 권한과 마스터 로그인 권한을 분리하는 것, 개인정보 활용 동의하지 않은 경우에 대한 처리 등 솔루션에 대한 의견을 제시하며 팀이 문제를 해결하는 것에 몰입합니다.
성과 & 배움
  • Master Login 기능 도입 후 2개월간 고객사 30% 이상 사용, Hotjar 설문 결과 만족도 4/5점 초과
  • 전체 제품에 적용하는 프로젝트로 사이드이팩트와 오류 없이 프로젝트 완료

Magic Link 기반 비로그인 접근 지원

역할: 비로그인 인증 설계 및 구현 전반
기술 스택: Python, Django, MySQL, Kafka, NHN SMS, SendGrid
프로젝트 개요
구성원의 가입 절차 없이 사용하길 원하는 고객사의 경우, 제품 사용이 어려웠습니다.
대규모 고객사의 니즈 증가로 비로그인 권한 인증을 구축하고, 특정 제품을 사용할 수 있도록 지원한 프로젝트입니다.
타 스쿼드의 업무를 지원한 프로젝트로 백엔드 개발은 혼자 진행했습니다.
문제 해결
  • 인증 로직은 토큰 기반으로 설계하였으며, 유효 시간 제한, 토큰 취소 처리, 특정 엔드포인트에만 접근 허용 등 보안 요구사항을 고려한 정책을 함께 구현했습니다.
    • 인증 처리 미들웨어, 데코레이터 등 모든 구현에 단위 테스트를 작성했고, PoC와 빠른 리뷰를 통해 프로젝트 전반의 리스크를 관리했습니다.
  • Magic Link는 SMS와 이메일로 발송하도록 구성했습니다.
    • 비로그인 상황에서 접근성을 높이기 위해 SMS도 지원해야함을 제안했습니다. 고객이 겪게될 문제를 항상 고려합니다.
    • Outbox Pattern을 사용해 SMS/Email 발송 이벤트를 생성하고, Kafka 기반 비동기 처리하였습니다.
성과 & 배움
  • 엔터 프라이즈 기업 도입 및 도입 기업의 Magic Link 서베이 참여율 80% 상회
    • SMS 지원으로 해당 기업이 이전에 다른 플랫폼을 통해 서베이를 했을 때의 참여율을 상회하는 결과를 얻었습니다. 고객의 문제를 해결하기 위해 몰두하며, 제품의 퀄리티를 스콥 아웃하지 않습니다.
  • 본인의 스쿼드 업무 디스커버리 기간인 약 3주동안 타 스쿼드 업무를 지원해준 프로젝트입니다.팀 전체의 비즈니스 임팩트를 항상 고려하며, 잘 모르는 분야도 도전하고 성취하는 것을 즐깁니다.
  • 권한 인증과 관련된 코드 리뷰와 설계에 대한 피드백을 수시로 요청했습니다. 그 결과 문제가 될 지점들을 빠르게 발견하는 등 리스크를 관리하며 프로젝트를 마무리했습니다. 적극적인 소통과 개발자로서 위험을 관리하는 법을 배웠습니다.

프로젝트 개요

제품 내에서 특약 조건을 유연하게 수용할 수 있도록 구독 도메인을 고도화한 프로젝트입니다.
문제 해결
  • 고객사별 특약을 반영할 수 있도록 구독 로직을 재설계하고, 레거시를 청산하는 작업을 진행했습니다.
    • 처음 개발 이후 관리가 되지 않던 회색 지대였고, 확장을 고려하지 않은 레거시 또한 많았습니다. 그 중 FE와 강결합되어 있는 구조를 개선하고, 전략 패턴을 활용해 특약을 처리하도록 구현해 유지 보수가 쉽도록 구현했습니다.
    • 레거시로 인해 작업에 차질이 생기지 않도록 이슈를 레이징해 팀이 문제를 인식하는 것을 중요하게 생각했습니다.
  • 해당 프로젝트의 범위 외로 사내 시스템을 구현해 QA 담당자가 직접 다양한 구독 시나리오를 테스트할 수 있도록 간편하게 구성했습니다.
    • 당시 개발자가 직접 구독 상태를 만들어줘야했던 상태로, 프로젝트와 별개로 불편함을 개선했으며 이 또한 프로젝트 기간 내에 완료해 QA 진행을 도왔습니다.
    • Django Template을 활용한 MVC 패턴으로 개발했으며, Cursor 등 AI 기반 개발 보조 툴을 활용해 빠르게 구축했습니다.
성과 & 배움
  • 당시 1명의 백엔드 엔지니어에게 의존하던 회색 지대의 문제를 공감해 자발적으로 참여한 프로젝트입니다. 레거시 청산부터 구독 도메인 개선을 주도하고 배운 프로젝트입니다.
  • 기존에 수작업으로 진행되던 특약 기반의 결제 및 관리 프로세스를 제품 내에서 자동화함으로써, 운영 공수 대폭 절감 및 QA 효율화
  • 특약 구독을 제품에서 직접 지원함으로써, 계약부터 결제까지의 전체 플로우를 간소화함으로 업무 효율화
  • 불편함이나 비효율을 개선해 팀 전체의 업무 효율을 높이는 것을 지향합니다. 개발자의 역량으로 비용을 줄이고 팀 전체의 효율을 올릴 수 있음을 배웠습니다.

Solana(솔라나) 입출금 지원 및 입금 아키텍처 개선

역할: 설계 및 개발 리드 (아키텍처 설계, DB 설계, 구현 전반)
기술 스택: Typescript, NestJS, ElasticCache(Redis), BullMQ
프로젝트 개요
솔라나는 평균 TPS가 3000이 넘어가는 블록체인 플랫폼입니다.
한 transaction에는 여러 event가 발생하므로 블록 당 처리해야하는 event는 약 10,000건이 넘어갑니다.
기존 입금 처리 로직으로는 cron job이 일정 주기마다 쌓인 블록을 처리하는 방식으로 솔라나와 같이 처리량이 매우 많아진 사항을 처리할 수 없습니다.
솔라나의 높은 TPS와 이벤트 처리량을 감당하기 위해 병렬 처리 아키텍처로 전환했습니다.
문제 해결
  • 두 가지 방안을 비교했으며, 최종적으로 BullMQ를 활용해 블록을 병렬 처리하도록 구현했습니다.
    • 모듈러 연산 기반 Job 분산: 처리할 블록을 job 단위로 분할하는 방식으로, 구현이 단순하고 기존 아키텍처와 유사하다는 장점이 있습니다. 하지만 DB와의 결합도가 높고 job 수 변경 시 도메인과 DB 양쪽에 수정이 필요하며, 특정 job 실패 시 해당 모듈러에 속한 블록 전체 처리가 지연된다는 단점이 있습니다.
    • Message Queue 기반 처리 (채택): 아키텍처 변경에 대한 고려가 더 필요했지만, 확장성과 장애 대응이 유연하다는 장점이 있습니다. 실패한 작업만 재처리할 수 있다는 것 또한 큰 장점입니다.
  • 이미 사용 중인 ElasticCache를 기반으로 구축해 인프라 부담을 줄였으며, 처리할 블록 정보를 가져오는 통신 레이어를 추상화해 의존을 역전 시켜 향후 인프라 변경에 용이하도록 설계했습니다.
성과 & 배움
  • Message Queue 기반 병렬 처리 구조로 전환하는 비교적 큰 변화였음에도 2주 내에 완료
    • 초기 설계 단계에서 입금 도메인과 블록 처리 도메인을 명확히 분리해 두었기 때문에, MQ 도입 시에도 최소한의 수정으로 전환
  • 기존 로직에서는 블록 처리 지연으로 서비스가 불가능했지만, 개선 후에는 블록 생성 후 1초 이내 입금 반영이 가능한 수준으로 개선

Multi-Party Computation Based Threshold Signature

역할: 논문 & 오픈 소스 리서치, 제품 PoC 설계 및 구현, 특허 등록
기술 스택: Rust, Rocket, Node.js, Express, MySQL
프로젝트 개요
Multi-Party Computation(MPC)은 연산에 참여하는 각 party의 input이 들어나지 않은 채로 모든 party가 동일한 결과들 얻도록 하는 연산 방식입니다.
지갑 서비스에서 가장 중요한 비밀값은 private key입니다. 대게의 경우 multi-sig를 사용해 private key를 분산 보관합니다. 하지만 서명하거나 키를 생성할 때 private key 원형이 노출되는 single point of failure가 존재한다는 단점이 있습니다.
이런 보안 이슈를 극복하기 위해 MPC를 활용합니다.
문제 해결
  • 논문과 오픈 소스를 리서치하고, MPC 연산을 사용해 블록체인 트랜잭션에 서명하고 전송하는 테스트 진행했습니다.
  • Rust Rocket의 오픈 소스를 기반으로 2 party 형식의 peer-2-peer 서버를 구현하고, 지갑 서비스와 연동해 제품에 적용까지 PoC를 마무리했습니다.
성과 & 배움
  • 리서치부터 오픈소스 테스트와 제품 설계까지 모든 과정을 주도한 프로젝트로 실질적인 문제 해결 경험을 쌓았습니다.
